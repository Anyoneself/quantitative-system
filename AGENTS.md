# AGENT 开发规范

## 总原则

- 使用 Python 作为主要开发语言。
- 优先完成最小可用实现，不做冗余设计。
- 不提前抽象，不为未来可能出现的需求增加复杂结构。
- 代码要清晰、直接、可测试。
- 所有代码注释必须使用中文。
- 注释只解释必要的业务规则、边界条件或非显然逻辑，避免描述代码本身。
- 保持模块职责单一，避免一个文件承担过多功能。
- 不引入项目当前不需要的框架、服务或中间件。

## 实现风格

- 能用函数解决的问题，不强行使用类。
- 能用简单类解决的问题，不设计复杂继承体系。
- 能用本地文件完成的数据存储，不提前引入数据库。
- 能用同步流程完成的任务，不提前引入异步、队列或分布式调度。
- 能用标准库解决的问题，不优先引入第三方依赖。
- 新增依赖必须有明确用途，并优先选择成熟、轻量、维护良好的库。
- 不写“万能工具函数”，只为当前真实需求提取公共逻辑。
- 不写空壳模块、空壳接口或暂时不会被调用的代码。

## 目录与模块

- 文档放在 `doc/`。
- 源码后续统一放在 `src/quant_system/`。
- 脚本后续统一放在 `scripts/`。
- 测试后续统一放在 `tests/`。
- 数据文件不提交到仓库，除非是小型测试样例。
- 配置文件与代码分离，敏感信息不得提交。

推荐最小目录：

```text
quantitative-system/
├── AGENT.md
├── README.md
├── doc/
├── src/
│   └── quant_system/
├── scripts/
└── tests/
```

## Python 代码规范

- 使用明确的变量名和函数名。
- 函数尽量短小，单个函数只做一件事。
- 参数数量过多时，优先检查设计是否过度复杂。
- 对外部输入做必要校验。
- 对文件路径、日期、股票代码等关键字段使用明确命名。
- 避免全局可变状态。
- 避免隐藏副作用，函数名应能表达是否会读取文件、写入文件或请求网络。
- 优先返回结构化数据，不返回难以解析的字符串。
- 错误处理要明确，不吞掉异常。

## 注释规范

- 所有注释使用中文。
- 不写无意义注释。
- 不写与代码重复的注释。
- 对以下场景必须写中文注释：
  - 行情数据字段映射。
  - 复权、停牌、涨跌停等交易规则处理。
  - 回测中的成交假设、手续费、滑点和调仓延迟。
  - 风控阈值的业务含义。
  - 为避免未来函数、幸存者偏差所做的处理。

## 数据采集规范

- 当前只使用免登录公开行情接口作为数据来源。
- 只采集公开可访问或正式授权的数据。
- 不绕过登录、付费、验证码、加密、访问控制或反爬限制。
- 采集逻辑必须包含超时、限速和有限重试。
- 原始数据先落盘，再进行清洗和标准化。
- 解析逻辑与请求逻辑分离。
- 页面或接口结构变化时，应通过测试或数据质量检查尽早发现。

## 量化系统规范

- 策略逻辑与数据读取分离。
- 回测逻辑与实盘交易逻辑分离。
- 策略只生成信号或目标仓位，不直接下单。
- 下单前必须经过风控检查。
- 回测必须考虑交易成本、滑点、停牌、涨跌停和调仓延迟。
- 不允许在回测中使用未来数据。
- 不允许只使用当前仍上市股票做历史回测。
- 每个策略都应能输出可复盘的交易记录和净值结果。

## 测试要求

- 新增核心逻辑时，应补充对应测试。
- 优先测试数据清洗、因子计算、回测成交、风控规则等容易出错的部分。
- 测试数据要小而明确。
- 测试应尽量稳定，不依赖实时网络。
- 网络采集相关测试应优先测试解析函数和字段映射。

## 文档要求

- 架构、设计和决策文档放在 `doc/`。
- 文档使用中文。
- 文档要服务实现，不写空泛概念。
- 当实现与文档不一致时，应同步更新文档。

## 禁止事项

- 禁止为了“看起来完整”创建大量空文件。
- 禁止提前引入微服务、消息队列、复杂插件系统或多层抽象。
- 禁止在没有真实需求时设计通用框架。
- 禁止提交密钥、Token、账号密码等敏感信息。
- 禁止把大体量行情数据直接提交到 Git。
- 禁止在注释中使用英文说明业务逻辑。
- 禁止静默忽略数据异常。

## 开发优先级

当前阶段优先顺序：

1. 项目结构清晰。
2. 公开行情数据采集最小闭环。
3. 数据清洗和质量校验。
4. 基础因子计算。
5. 最小回测引擎。
6. 简单策略验证。
7. 风控规则。
8. 模拟交易。
9. 实盘交易。

任何开发任务都应先确认是否服务当前阶段目标。若不是，默认暂缓。
